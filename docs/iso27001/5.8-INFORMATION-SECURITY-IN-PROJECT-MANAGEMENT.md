# Information Security in Project Management
**ISO/IEC 27001:2022 — Control 5.8**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Ensure information security is integrated into all phases of project and feature development, not added as an afterthought.

## Security Gates in Development Lifecycle

### 1. Design / Planning
- New features involving user data or external integrations require a threat model update (`THREAT_MODEL.md`).
- Authentication and authorization requirements must be defined before implementation begins.
- Roles and access levels must be specified in the task/issue.

### 2. Implementation
- Follow secure coding practices in `docs/BACKEND_DEVELOPER_GUIDE.md`.
- All new API endpoints must have `@UseGuards(JwtAuthGuard, RolesGuard)` applied.
- Input DTOs must use `class-validator` decorators.
- No secrets or credentials in code — environment variables only.

### 3. Code Review
- Security-relevant changes (auth, guards, database queries, webhook handlers) require review by Technical Lead.
- Check for: missing guards, raw SQL, hardcoded values, missing input validation.

### 4. Testing
- Run `npm audit` before merging to main.
- Verify authentication is enforced on all new routes.
- Confirm new migrations do not expose sensitive data.

### 5. Deployment
- Environment variables validated by `validateEnvironment()` on startup.
- Production deployments require confirmation that `.env.production` is not committed to Git.
- Post-deployment: verify `/api/v1/core/health` responds correctly.

## Change Management
All changes to production configuration follow the `src/CHANGE_MANAGEMENT_POLICY.md` process.
