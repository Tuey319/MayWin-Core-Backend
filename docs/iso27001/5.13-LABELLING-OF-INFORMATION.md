# Labelling of Information
**ISO/IEC 27001:2022 — Control 5.13**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Ensure information assets are labelled consistently with their classification to support correct handling.

## Labelling Approach

### Code and Configuration Files
| Asset Type | Label Convention | Example |
|---|---|---|
| Environment files | Filename prefix | `.env.production`, `.env.staging`, `.env` (dev) |
| Secret fields in code | Comments | `// SECRET — never log` |
| Policy documents | Control reference in header | `ISO/IEC 27001:2022 — Control 5.x` |
| Database columns storing sensitive data | Entity comments | `/** Raw 6-digit code (short-lived) */` on `otp_code` |

### Database
- Tables storing personal data are prefixed with their schema (`maywin_db.*`) and documented in entity files under `src/database/entities/`.
- Fields containing personal data are annotated in `5.12-CLASSIFICATION-OF-INFORMATION.md`.

### API Responses
- Sensitive fields (e.g. `password_hash`, `otp_code`) are **never** included in API response DTOs — services return only mapped view objects.
- Error responses via `HttpExceptionFilter` never include internal field names or stack traces.

### Documents
All ISO 27001 compliance documents in `docs/iso27001/` include the control number and classification level in their header.

## Review
Labels and classification assignments are reviewed annually or when the data model changes significantly.
