# Information Transfer
**ISO/IEC 27001:2022 — Control 5.14**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Establish rules and controls for the secure transfer of information within the platform and to external parties.

## Transfer Channels and Controls

### Client ↔ API (HTTPS)
- All API traffic is over HTTPS only. `Strict-Transport-Security` header enforced via `helmet()`.
- CORS restricted to origins listed in `ALLOWED_ORIGINS` environment variable.
- No sensitive data in URL query parameters — credentials and tokens passed via request body or `Authorization` header only.

### Backend ↔ Database (AWS RDS)
- Connection uses `PGSSLMODE=require` — TLS enforced.
- Credentials stored in environment variables, never hardcoded.

### Backend ↔ AWS Services (S3, Step Functions, Lambda)
- All AWS SDK calls use IAM role credentials via environment variables.
- S3 artifacts stored in a private bucket — no public ACLs.
- Step Functions input/output is sanitized — no raw credentials in execution payloads.

### Backend ↔ LINE Messaging API
- All communication over HTTPS to `api.line.me`.
- Webhook requests verified using HMAC-SHA256 signature before processing.
- LINE user IDs are not exposed in API responses to unauthorized users.

### Backend ↔ Gemini AI API
- All requests over HTTPS.
- API keys stored in environment variables only.
- No nurse personal data (names, schedules) included in AI prompts beyond what is operationally necessary.

### Email (OTP Delivery)
- OTPs sent via SMTP/TLS using `nodemailer`.
- OTP codes are never logged and expire after 10 minutes.
- Failed delivery causes immediate `ServiceUnavailableException` — no fallback logging.

## Prohibited Transfers
- No export of nurse personal data to personal email addresses or unapproved third-party services.
- No transmission of credentials or API keys via messaging applications (Slack, LINE, email).
