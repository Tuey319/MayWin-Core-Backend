# Protection of Test Information
**ISO/IEC 27001:2022 — Control 8.33**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Protect operational data used in testing to prevent unauthorized disclosure and ensure test environments do not expose production data.

## Test Environment Policy

### Principle: No Production Data in Testing
- Test environments must use synthetic or anonymized data only.
- Seed scripts in `src/database/seeds/` provide realistic non-production data.
- JSON seed files (`data/test_60nurses_jan2026_business_only_availability.json`) use synthetic nurse names and schedules.
- Under no circumstances should a production RDS snapshot be restored to a development or staging environment without anonymization.

### Test Data Sources
| Source | Usage | Data Type |
|---|---|---|
| `src/database/seeds/seed-from-json.ts` | Load test data from JSON files | Synthetic nurse and schedule data |
| `src/database/seeds/april-2026-seed.ts` | April 2026 test scenario | Synthetic |
| `data/*.json` | Bulk test nurse data | Synthetic — no real personal data |

### Test Accounts
- Test user accounts created in staging use `@example.com` email addresses.
- Test passwords meet the same complexity requirements as production (enforced by `SignupDto` validation).
- Test API keys (Gemini, LINE) use sandbox/test credentials — never production keys.

## Test Isolation
- `jest.config.js` configures test runner against isolated test database.
- Integration tests in `test/` directory connect to a local or staging database — never production.
- Environment variable `NODE_ENV=test` distinguishes test context.

## Cleanup
Test data is removed after test execution. Staging database is reset periodically using seed scripts.
