# Use of Privileged Utility Programs
**ISO/IEC 27001:2022 — Control 8.18**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Restrict and monitor the use of utility programs with privileged capabilities that could affect the system.

## Privileged Utilities in Use

| Tool | Privilege Level | Who Can Use | Controls |
|---|---|---|---|
| TypeORM CLI (`npm run migration:run`) | Database schema changes | Technical Lead only | Must be run against staging first; manual RDS snapshot required before production run |
| AWS CLI / Console | Infrastructure access | Technical Lead only | IAM permissions scoped; MFA required on AWS account |
| `psql` direct database access | Full database access | Technical Lead only | Only via AWS RDS proxy or bastion; all queries logged |
| Seed scripts (`src/database/seeds/`) | Bulk data insert | Developers (staging only) | Must never be run against production database |
| `restore-db.sh` | Database restore | Technical Lead only | Run only during incident recovery; log the action |
| `backup-db.sh` | Database export | Technical Lead + ORG_ADMIN | Log the action in incident/maintenance log |

## Controls
1. **Principle of least privilege**: No developer has direct production database access. Database credentials are held by the application and Technical Lead only.
2. **Approval required**: Any direct database modification (outside normal application operation) requires Technical Lead sign-off.
3. **Audit trail**: All migration runs are recorded in TypeORM migration history table. Direct database changes must be manually logged.
4. **Staging first**: All migration scripts and seed scripts must be tested in staging before production.
5. **No `DROP TABLE` or `DELETE` without backup**: Always snapshot RDS before any destructive operation.

## Prohibited Actions
- Running seed scripts against the production database.
- Using `typeorm-ts-node-commonjs` CLI directly against production without Technical Lead approval.
- Granting any application role `SUPERUSER` privileges in PostgreSQL.
