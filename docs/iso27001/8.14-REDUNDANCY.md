# Redundancy of Information Processing Facilities
**ISO/IEC 27001:2022 — Control 8.14**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Ensure sufficient redundancy in information processing facilities to meet availability requirements.

## Redundancy Architecture

### Application Layer
| Component | Redundancy | Notes |
|---|---|---|
| NestJS API (Lambda) | AWS Lambda auto-scales, multi-AZ by default | No single point of failure |
| NestJS API (ECS/EC2) | Configure ECS service with min 2 tasks across AZs | Recommended for production |
| Step Functions | AWS-managed, multi-AZ | Solver orchestration |

### Data Layer
| Component | Redundancy | Notes |
|---|---|---|
| PostgreSQL (RDS) | Multi-AZ deployment recommended | Automatic failover in ~60 seconds |
| S3 | 99.999999999% durability — automatic replication | No additional config needed |

### Network Layer
| Component | Redundancy | Notes |
|---|---|---|
| AWS Region | ap-southeast-1 (3 availability zones) | Use Multi-AZ for database and ECS |
| Load Balancer | AWS ALB with health checks | Required for ECS deployment |

## Current Gaps
- RDS Multi-AZ: Recommend enabling for production deployment.
- ECS minimum task count: Ensure at least 2 tasks running across different AZs.
- Health endpoint: `GET /api/v1/core/health` exists and returns system status — used by load balancer health checks.

## Availability Target
| Service | Target Availability |
|---|---|
| API | 99.9% (three nines) |
| Database | 99.95% (with Multi-AZ) |
| LINE Bot webhook | 99.9% |

## Failover Testing
Test Multi-AZ failover annually by simulating an AZ failure in a staging environment.
