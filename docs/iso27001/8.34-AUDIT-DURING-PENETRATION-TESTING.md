# Protection of Information Systems During Audit Testing
**ISO/IEC 27001:2022 — Control 8.34**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Protect the MayWin system during audit and penetration testing activities to prevent disruption or data exposure.

## Audit and Testing Activities

### Academic Security Audit (This Repository)
- Auditors have read-only access to the source code repository.
- No access to production environment, database, or runtime logs.
- Audit performed against source code and documentation only.

### Penetration Testing (When Conducted)
Before any penetration test:
1. **Written authorization**: Obtain written approval from the Technical Lead and project owner.
2. **Scope definition**: Define exact scope — IP ranges, API endpoints, test accounts only.
3. **Time window**: Schedule during low-usage hours to minimize impact on nurses.
4. **Test environment preferred**: Conduct testing against staging environment, not production.
5. **No data exfiltration**: Testers must not retain or exfiltrate nurse personal data discovered during testing.
6. **Credential isolation**: Create dedicated test accounts — do not use production admin credentials.

### During Testing
- Monitor AWS CloudWatch and application logs for unusual activity.
- Audit logs capture all actions performed by test accounts.
- Rate limiting (`ThrottlerModule`) remains active during testing — testers must account for this.

### After Testing
- Review and remediate all findings within agreed timescales (Critical: 48 hours, High: 7 days, Medium: 30 days).
- Deactivate test accounts.
- Document findings and fixes in `5.27-LEARNING-FROM-INCIDENTS.md`.
- Rotate any credentials used during the test.

## Responsible Disclosure
If a security researcher discovers a vulnerability:
1. Report via GitHub Security Advisory (private disclosure).
2. Technical Lead will acknowledge within 48 hours.
3. Fix will be implemented and deployed within 30 days for critical issues.
