# Installation of Software on Operational Systems
**ISO/IEC 27001:2022 — Control 8.19**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Control the installation of software on operational (production) systems to prevent unauthorized or untested software from being deployed.

## Deployment Process

### Application Deployment
1. All changes merged to `main` branch via Pull Request with at minimum 1 review.
2. Build process: `npm run build` — TypeScript compiled to `dist/`.
3. Tests must pass: `npm test`.
4. `npm audit` run — no critical vulnerabilities allowed.
5. Deployment to production via AWS Lambda deployment package or ECS task update.
6. Environment variables injected via AWS Secrets Manager / Parameter Store — no `.env` files in production.

### Dependency Installation
- Only packages listed in `package.json` may be installed in production.
- `npm ci` used in CI/CD (not `npm install`) — ensures exact versions from `package-lock.json`.
- No `--force` flag when installing packages.
- New dependencies require Technical Lead review before merging.

### Database Changes
- Only through versioned TypeORM migration files in `src/database/migrations/`.
- No ad-hoc schema changes in production.
- Migration files are immutable once merged to `main`.

### Prohibited in Production
- `devDependencies` packages (jest, ts-node, etc.) must not be deployed to production Lambda/ECS.
- No direct file system writes to application directories at runtime.
- No execution of dynamically generated code (`eval`, `new Function`).

## Change Management
All deployment activities follow `src/CHANGE_MANAGEMENT_POLICY.md`.
