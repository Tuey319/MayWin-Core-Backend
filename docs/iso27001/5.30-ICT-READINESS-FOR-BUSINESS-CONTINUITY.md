# ICT Readiness for Business Continuity
**ISO/IEC 27001:2022 — Control 5.30**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Ensure ICT systems can support business operations during and after a disruption.

## Recovery Objectives

| Component | RPO (Recovery Point Objective) | RTO (Recovery Time Objective) |
|---|---|---|
| PostgreSQL (RDS) | 24 hours (daily automated backup) | 2 hours |
| Application (Lambda/ECS) | 0 (stateless — redeploy from Git) | 30 minutes |
| S3 artifacts | 0 (cross-region replication optional) | Immediate |
| LINE Bot integration | 0 (stateless webhook) | 30 minutes (credential rotation) |

## Backup Strategy
- **RDS**: Automated daily snapshots enabled. Retention: 7 days. Manual snapshot before every major migration.
- **S3**: Versioning enabled on artifacts bucket. No deletion policy — artifacts are immutable.
- **Application code**: Git repository on GitHub — complete history maintained.
- **Environment configuration**: Documented process for re-injecting environment variables into AWS Lambda/ECS from secure storage (AWS Secrets Manager or Parameter Store).

## Recovery Procedure
1. **Database failure**: Restore from most recent RDS snapshot. Run pending TypeORM migrations (`npm run migration:run`).
2. **Application failure**: Redeploy from Git main branch. No data loss (stateless application).
3. **Credential compromise**: Follow `INCIDENT_RESPONSE_PLAN.md` — rotate all affected secrets, redeploy.
4. **AWS region outage**: Re-provision in alternative region using infrastructure-as-code scripts. RDS restore from cross-region snapshot.

## Testing
Business continuity recovery procedures should be tested annually. At minimum, verify:
- RDS restore completes within RTO.
- Application deploys cleanly from a fresh Git clone.
- Environment variable injection works correctly.
