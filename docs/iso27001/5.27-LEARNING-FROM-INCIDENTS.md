# Learning from Information Security Incidents
**ISO/IEC 27001:2022 — Control 5.27**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Use lessons learned from security incidents to improve controls and reduce the likelihood of recurrence.

## Post-Incident Review Process

After any security incident is resolved:

1. **Within 48 hours**: Technical Lead conducts a post-incident review.
2. **Document**: Record the root cause, timeline, impact, and resolution in the incident log.
3. **Identify gaps**: Determine which control failed or was absent.
4. **Update controls**: Implement code, configuration, or policy changes to address the gap.
5. **Update documentation**: Revise the relevant policy document and `THREAT_MODEL.md` if needed.
6. **Share learnings**: Brief the development team on what changed and why.

## Known Historical Lessons Applied to This Codebase

| Lesson | Control Applied |
|---|---|
| OTP bypass via environment variable left in production | Removed `AUTH_DISABLE_OTP` bypass entirely |
| OTP logged in plaintext as dev fallback | Removed fallback; SMTP failure now throws hard error |
| `RolesGuard` returning `true` — no RBAC enforcement | Implemented `Reflector`-based role checking |
| `Math.random()` for security tokens | Replaced with `crypto.randomBytes` / `crypto.randomInt` |
| Webhook signature skipped if secret missing | Changed to fail-closed: reject all requests if secret absent |
| Stack traces leaking in error responses | `HttpExceptionFilter` sanitises all error responses |
| No rate limiting on login/OTP | `@Throttle` decorator on auth endpoints |
| Credentials committed in `.env.production` | Rotated all credentials; `.env.*` added to `.gitignore` |

## Incident Register Location
Incidents are recorded in `INCIDENT_RESPONSE_PLAN.md` and the audit log (`AuditLogsService`).
