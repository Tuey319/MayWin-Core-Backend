# Outsourced Development
**ISO/IEC 27001:2022 — Control 8.30**
**MayWin Nurse Scheduling Platform**
Last reviewed: 2026-03-30

---

## Purpose
Ensure that outsourced or externally contributed development activities are managed with the same security requirements as internal development.

## Current Development Model
The MayWin backend is developed by a team of student developers as part of an academic project. All developers work under the supervision of a Technical Lead.

## Security Requirements for All Contributors

### Before Access is Granted
- GitHub account must have 2FA enabled.
- Developer must sign or acknowledge the NDA / Acceptable Use Policy.
- Access granted to repository — no direct production access.

### During Development
- All code must follow the secure coding standards in `docs/BACKEND_DEVELOPER_GUIDE.md`.
- No developer may commit secrets, credentials, or production environment variables.
- All contributions via Pull Request — no direct push to `main`.
- Technical Lead reviews all security-relevant changes.

### Code Quality and Security Checks
- `npm audit` must pass (no critical/high vulnerabilities) before merge.
- TypeScript compilation must pass (`npx tsc --noEmit`).
- Test suite must pass (`npm test`).

### Intellectual Property
- All code produced is owned by the project/institution per the terms of the academic engagement.
- No open-source code may be incorporated without license compatibility review (see `5.32-INTELLECTUAL-PROPERTY-RIGHTS.md`).

### Off-boarding
See `5.11-RETURN-OF-ASSETS.md` and `6.5-RESPONSIBILITIES-AFTER-TERMINATION.md`.

## Third-Party Code Components
All third-party libraries are managed via `npm` and tracked in `package.json` and `package-lock.json`. See `5.21-MANAGING-SECURITY-IN-ICT-SUPPLY-CHAIN.md`.
