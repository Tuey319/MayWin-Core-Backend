# Privacy Policy

**Control**: ISO/IEC 27002 A.5.34, A.8.26

This policy outlines how the MayWin Core Backend processes, protects, and manages Personal Data in compliance with the Thailand Personal Data Protection Act (PDPA) B.E. 2562.

## 1. Scope
This policy applies to all personal and sensitive data processed by the MayWin Core Backend, including data stored in the PostgreSQL database and data handled during API interactions.

## 2. Personal Data Processed
The backend processes the following categories of personal data to provide its nurse scheduling services:

| Data Category | Description | Example Evidence |
| :--- | :--- | :--- |
| **User Account Data** | Information required to identify and authenticate users. This includes name, email, and assigned roles. | `src/user.entity.ts` |
| **Authentication Data** | Hashed passwords and JWTs used to secure user sessions. Plaintext passwords are never stored. | `src/auth.service.ts` (bcrypt), `src/auth.module.ts` (JWT) |
| **Scheduling Data** | Worker assignments to specific shifts on specific dates. This links a worker's identity to their work schedule. | `SchedulesModule`, `JobsModule` |
| **Availability & Preferences** | Worker-submitted data on their availability, days off, and shift preferences. | `AvailabilityModule`, `WorkerPreferencesModule` |
| **Audit & Security Logs** | System-generated logs containing user identifiers (User ID) and network identifiers (IP address) for security monitoring. | `src/common/interceptors/security-logger.interceptor.ts` |
| **Worker Messages** | Internal messages between workers and managers, which may contain personal information. | `WorkerMessagesModule` |

## 3. PDPA Compliance Principles

### 3.1. Lawfulness, Fairness, and Transparency
We process personal data based on the necessity to perform the scheduling service contract. This policy provides transparency into our data handling practices.

### 3.2. Purpose Limitation
Personal data is collected and processed solely for the purpose of workforce management, scheduling, and ensuring the security of the platform. Data is not used for any other purpose without consent.

### 3.3. Data Minimization
We collect only the minimum personal data necessary to create and manage schedules.
- **Evidence**: The `ValidationPipe` with `whitelist: true` and `forbidNonWhitelisted: true` (`src/main.ts`) ensures that only data defined in DTOs is processed, preventing extraneous data collection.

### 3.4. Accuracy
Users are provided with interfaces (via the frontend) to review and update their personal information to ensure it remains accurate.

### 3.5. Storage Limitation
Data is retained only for as long as necessary.
- **Evidence**: See `docs/security/DATA_RETENTION_POLICY.md` for specific retention periods.

### 3.6. Integrity and Confidentiality (Security)
We implement robust technical and organizational measures to protect personal data.
- **Access Control**: All API endpoints are protected by default with `JwtAuthGuard`. Access to data is restricted based on user roles (`NURSE`, `UNIT_MANAGER`, `ORG_ADMIN`) via the `RolesGuard`. (ISO/IEC 27002 A.5.15)
- **Encryption**:
  - **In Transit**: All API communication is secured with TLS. (`NETWORK_SECURITY.md`)
  - **At Rest**: Passwords are cryptographically hashed using `bcrypt`. (`src/auth.service.ts`) (ISO/IEC 27002 A.8.24)
- **Secrets Management**: All secrets, including database credentials and JWT secrets, are managed via environment variables and are never hardcoded. (`ENVIRONMENT_POLICY.md`) (ISO/IEC 27002 A.5.1)

### 3.7. Accountability
Security roles and responsibilities are clearly defined to ensure accountability for data protection.
- **Evidence**: See `docs/security/SECURITY_ROLES.md`. Logs are generated by the `SecurityLoggerInterceptor` to track access and changes to data. (ISO/IEC 27002 A.8.15)

## 4. Data Subject Rights
The backend architecture supports the fulfillment of data subject rights as required by PDPA, including the right to access, correct, or request deletion of their personal data, facilitated through API endpoints.
